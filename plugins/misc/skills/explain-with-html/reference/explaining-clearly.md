# How to make an explanation easy to understand

Before turning a topic into HTML, decide **"whose gap am I closing"** rather than "what should I write".
Design in the order below, then write. These are practical distillations of repeatedly-validated
principles from learning science (cognitive load theory, multimedia learning, advance organizers,
conceptual change).

## 1. Model the reader — separate what they know from what they don't

The reader is usually **the user in this conversation**. The transcript is your best signal.

- **What they already know**: terms the user used correctly, systems they built or referenced,
  questions that presuppose knowledge. Use what they know as **stepping stones** — don't re-teach it.
- **What they don't know / the real sticking point**: the actual question asked, the misconception
  behind it, the parts they re-ask. An "explain it more simply again" request usually means they
  have a model that didn't quite click.
- **Why now**: the request itself reveals the gap. Aim the document at **closing that one gap**,
  not at being an encyclopedia entry.
- **Beware the curse of knowledge**: the step an expert skips as obvious is usually exactly where an
  ordinary reader is stuck. Don't skip it.
- If only a topic is given with no conversational signal, assume a reasonable audience and state it
  in one line at the top ("This assumes a reader who knows X").

## 2. Entry point and depth

- Connect **known → unknown**. Anchor each new concept to something the reader already holds.
- **Calibrate depth**: keep the familiar parts short; spend the words on the **one hard part**.
- **One concept at a time.** Working memory is narrow (a few chunks). Don't dump new elements at
  once — chunk and stack them in order.

## 3. Sequence

- **Big picture first, then the machinery, then the exceptions.** Let the reader hold a map before
  diving into detail (advance organizer).
- **Motivation before mechanism.** Open with "why is this needed / what problem does it solve" to
  set a hook.
- **Concrete before abstract.** Show an example with real values, or an analogy, first — then generalize.

## 4. Devices that aid understanding

- **Analogy**: map the new concept onto something familiar (e.g. a lock → an "in-use sign"). But
  point out **where the analogy breaks** so it isn't over-extended.
- **Refute misconceptions head-on**: "it's tempting to think X, but actually Y". Deny the wrong model
  first, then give the right one — it sticks better than just stating the answer (conceptual change).
- **Words and pictures together.** Use diagrams (mermaid) for flow/structure/state. Put labels
  **inside** the picture, and place the picture **right next to** the sentence it explains
  (avoid split attention).
- **Worked example**: run it once with real values.
- **Cut the tangents.** Drop interesting-but-irrelevant detail. Every element must serve closing the
  gap (coherence principle).
- **Signal the structure**: a summary box, section headings that state the question that section answers.

## 5. Wrap-up

- **Recap in one line.**
- **State the boundary**: what this does *not* cover, where it stops. Prevents over-extending the
  analogy or model.
